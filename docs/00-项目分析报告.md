## 1. 需求分析

### 1.1 项目本质

这是一个**个人开发工作台（Personal Developer Workstation）**，而非单一工具。核心特征是：

- **长期演进**：未来数年持续增加模块
- **模块化**：每个功能独立，可单独开发/测试/启用
- **桌面 GUI**：最终形态是真正的 Windows 桌面软件
- **教育优先**：开发过程本身就是学习过程

### 1.2 功能需求拆解

| 阶段 | 模块 | 核心功能 | 优先级 |
|------|------|----------|--------|
| Phase 1 | Health | 开发环境一键健康检查 | P0 |
| Phase 1 | Update | 工具版本检查与更新提示 | P0 |
| Phase 1 | Clean | Downloads 智能分类整理 | P1 |
| Phase 1 | New | 项目模板创建器 | P0 |
| Phase 1 | Study | 官方文档下载与学习管理 | P1 |
| Phase 1 | GitHub | 仓库管理、Issue、PR | P0 |
| Phase 1 | Browser | Playwright 浏览器自动化 | P1 |

| Phase 2 | — | 统一模块接口抽象 | — |
| Phase 3 | — | GUI 层 | — |
| Phase 4 | — | UX 优化、深色模式、图标 | — |
| Phase 5 | — | 打包 .exe | — |

### 1.3 非功能需求

- 现代化 UI（深色模式、统一设计语言）

## 2. 需求评估与建议

### 2.1 合理性评估：✅ 非常合理

这个项目定位精准，原因：

1. **真实需求驱动**：你确实需要管理开发环境、GitHub、文档等，不是凭空想象
2. **难度梯度合理**：CLI → 抽象 → GUI → 打包，每一步都建立在前面基础上
3. **规模可控**：8 个初始模块，每个都可以在几天内完成 MVP
4. **学习价值极高**：涵盖模块化设计、GUI 开发、打包分发、持续维护

### 2.2 我的建议

#### 建议 1：Phase 1 先做 4 个核心模块，再做其余 4 个

理由：8 个模块同时开发会让架构设计压力过大。建议先做 **Health、New、GitHub、Backup** 这 4 个最核心的，跑通完整流程（模块→接口→测试→文档），再复制模式做其余 4 个。这样：

- 前 4 个模块建立「模板模式」
- 后 4 个模块验证「可扩展性」
- 如果架构有问题，在第 4 个模块就能发现，而不是第 8 个

#### 建议 2：每个模块都是一个独立的 Python 包

不要把所有代码堆在一起。每个模块应该是：

```
src/modules/health/
├── __init__.py
├── module.py       # 继承 BaseModule
├── checker.py      # 具体检查逻辑
├── reporter.py     # 结果格式化
└── __main__.py     # 可独立运行: python -m modules.health
```

这样做的好处：
- 每个模块可以独立运行（`python -m modules.health`）
- 每个模块可以独立测试
- 模块之间通过 EventBus 松耦合
- 未来可以独立发布/禁用

#### 建议 3：配置管理用 JSON Schema 验证

不要用裸 dict。用 dataclass + JSON Schema：

```python
@dataclass
class HealthConfig:
    checks: List[str]       # 要运行的检查项
    thresholds: Dict[str, int]  # 阈值
    auto_fix: bool = False  # 是否自动修复
```

这样：
- IDE 有自动补全
- 类型安全
- 配置文件可以被验证
- 未来 GUI 可以自动生成设置表单

#### 建议 4：现在就为 i18n（国际化）做准备

虽然目前只用中文，但用 `_("text")` 包裹所有用户可见字符串：

```python
def _(text: str) -> str:
    return text  # 未来可以从 .po/.json 加载翻译
```

这是一个好习惯，改动成本几乎为零，但未来如果要加英文界面就很容易。

#### 建议 5：Git 仓库现在就初始化

从第一个文件开始就用 Git 管理。建议的仓库结构：

- 主仓库：`YQDevAssistant`（本项目）
- 分支策略：`main`（稳定） + `develop`（开发） + `feature/xxx`（功能分支）
- 每个模块一个 feature 分支

### 4.3 最终推荐：PySide6 🏆

**推荐 PySide6（Qt for Python）**，理由：

#### 为什么不是 Flet？
Flet 虽然上手快，但：
- 2022 年才发布，API 仍在变化
- 社区较小，遇到问题难搜答案
- 底层是 Flutter，调试困难（中间隔了两层抽象）
- 对于「未来数年维护」的项目，稳定性风险大

#### 为什么是 PySide6？
1. **Qt 是工业标准**：Spotify、Adobe、Autodesk 都用 Qt
2. **30 年沉淀**：文档、社区、教程极其丰富
3. **Signal/Slot 机制**：这是学习事件驱动架构的最佳范本
4. **Qt Designer**：可以可视化拖拽设计界面，降低入门门槛
5. **QSS（Qt Style Sheets）**：用 CSS 语法定制主题，实现深色模式
6. **丰富的 Widget**：Tab、Tree、Table、WebView、Chart 等，满足所有需求
7. **LGPL 协议**：免费用于闭源商业软件
8. **`entry_points` 插件系统**：Python 标准，最适合模块化架构
9. **AI 对 Qt 代码训练极充分**：代码生成质量最高

#### 一个实际的对比

用 Flet 做一个「带侧边栏的多标签编辑器」：
```python

# 很多行为需要手动管理
```

用 PySide6 做同样的事：
```python

# 都是现成的成熟组件
```

对于你的项目（类似 IDE 的开发工作台），PySide6 的组件更适合。

### 5.1 分层架构

```
┌─────────────────────────────────────────────────┐
│                  UI Layer (PySide6)              │
│  MainWindow / SettingsPage / ModuleDashboards    │
├─────────────────────────────────────────────────┤
│                Service Layer (Core)              │
│  ModuleManager / ConfigManager / EventBus        │
│  LogManager / PluginLoader / UpdateChecker       │
├─────────────────────────────────────────────────┤
│                Module Layer (modules/)           │
│  Health / Update / Clean / New / Study /         │
│  GitHub / Browser / Backup / ...                 │
├─────────────────────────────────────────────────┤
│              Services Layer (services/)          │
│  GitHubService / PlaywrightService /             │
│  FileSystemService / NetworkService              │
├─────────────────────────────────────────────────┤
│                Utils Layer (utils/)              │
│  Shell / Validators / Formatters / Constants     │
├─────────────────────────────────────────────────┤
│                Data Layer                        │
│  config/*.json / logs/*.log / backups/           │
└─────────────────────────────────────────────────┘
```

### 5.2 核心设计模式

#### 模式 1：抽象基类（BaseModule）

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any

@dataclass
class ModuleResult:
    success: bool
    data: Any
    errors: list[str]
    warnings: list[str]
    duration_ms: float

class BaseModule(ABC):
    """所有模块的抽象基类"""
    
    # 模块元信息（子类必须定义）
    name: str
    version: str
    description: str
    icon: str = "📦"
    
    @abstractmethod
    def check(self) -> ModuleResult:
        """检查/诊断（只读，无副作用）"""
        ...
    
    @abstractmethod
    def execute(self, **kwargs) -> ModuleResult:
        """执行操作（可能有副作用）"""
        ...
    
    def configure(self, config: dict) -> None:
        """配置模块"""
        ...
    
    def get_status(self) -> dict:
        """获取模块状态"""
        ...
```

#### 模式 2：EventBus（事件总线）

模块之间不直接调用，而是通过 EventBus 通信：

```python
# 解耦示例
event_bus.publish("github:repo_cloned", {"name": "my-project"})
# → New 模块自动检测并提示"是否要打开此项目？"
# → Backup 模块自动在备份中包含新仓库
```

#### 模式 3：插件自动发现

```python
# 通过 Python entry_points 或目录扫描
# 新增模块只需：
# 1. 创建 src/modules/xxx/ 目录
# 2. 继承 BaseModule
# 3. 在 module.py 中导出 MODULE_CLASS
# 无需修改任何核心代码
```

### 5.3 数据流

```
用户操作 → UI → ModuleManager → 对应Module.check/execute()
                                    ↓
                               EventBus.publish(结果)
                                    ↓
                           LogManager 记录日志
                           ConfigManager 保存配置
                           UI 更新状态栏
```

### 每个目录为什么存在

| 目录 | 存在理由 |
|------|----------|
| `src/core/` | 框架代码。模块之间共享的基础设施。改动频率最低。 |
| `src/modules/` | 功能代码。每个模块独立目录，可单独开发/测试/禁用。 |
| `src/services/` | 外部服务封装。有状态的 API 包装器（GitHub API、Playwright 等），模块不直接调外部 API。 |
| `src/ui/` | 表现层代码。与业务逻辑分离，Phase 3 才大规模开发。 |
| `src/cli/` | 命令行入口。Phase 1-2 的主要交互方式。 |
| `src/utils/` | 无状态的工具函数。不依赖 core 或 modules。纯函数为主。 |
| `tests/` | 镜像 src 结构。一眼就能知道哪些代码有测试。 |
| `docs/` | 项目文档。代码即文档不够，需要架构决策记录。 |
| `config/` | 出厂默认配置。用户配置在运行时目录（如 `~/.yqa/`）。 |
| `assets/` | 图片、图标等二进制文件。不放在 src 里。 |
| `scripts/` | 自动化脚本。不属于产品代码。 |
| `logs/` | 运行时日志输出。不提交到 Git。 |
| `backups/` | Backup 模块的默认输出目录。不提交到 Git。 |

## 7. 可复用脚本评估

### 7.1 三个 Shell 脚本的价值

| 脚本 | 复用价值 | 处理方式 |
|------|----------|----------|
| `dev-check.sh` | ⭐⭐⭐⭐⭐ | 逻辑直接映射为 Health 模块的 checker.py |
| `weekly-health.sh` | ⭐⭐⭐⭐⭐ | 健康评分算法保留，整合到 Health 模块 |
| `dev-update-check.sh` | ⭐⭐⭐⭐ | 逻辑映射为 Update 模块的 version_checker.py |

### 7.2 为什么用 Python 重写而不是直接调用 Shell？

1. **跨平台**：Shell 脚本依赖 Git Bash，Python 在 Windows 原生运行
2. **可测试**：Python 代码可以写单元测试
3. **类型安全**：Python 类型注解减少错误
4. **错误处理**：Python 异常机制比 Shell 的 `set -e` 更可靠
5. **统一性**：所有模块使用同一语言，降低维护成本
6. **打包**：PyInstaller 打包 .exe 时可以包含 Python 依赖，但 Shell 脚本需要额外处理

### 7.3 `demo.mjs` 的价值

Playwright 的 JavaScript 演示代码保留在 `docs/references/` 中作为参考，但 Browser 模块用 Python 版 Playwright API（`playwright` Python 包）。

## 9. 第一阶段 Milestone（详细）

### M1.1: 项目基础设施 🎯

**预计时间**：1 周

**范围：仅做架构框架，不实现任何功能模块。**

这是整个项目最重要的阶段。我们在这里建立所有约定和基础设施，后续 8 个模块都以此为模板。

**架构决策记录（ADR）：**
- 每个重要决策都记录在 `docs/adr/` 中
- 包含：背景、决策、后果、备选方案

**交付物：**

| # | 任务 | 产出 | 为什么先做这个 |
|---|------|------|----------------|
| 1 | 创建 Git 仓库 + 推送到 GitHub | `github.com/lt3045502989-dev/YQDevAssistant` | 版本控制是一切的基础 |
| 2 | 搭建完整目录结构 | 所有目录 + `.gitkeep` | 统一认知，知道代码放哪里 |
| 3 | 编写 `pyproject.toml` | 项目元信息 + 依赖声明 | 现代 Python 项目标准 |
| 4 | 实现 `BaseModule` | 抽象基类 + `ModuleResult` | 所有模块的「宪法」，必须先定 |
| 5 | 实现 `ModuleManager` | 模块发现 + 注册 + 生命周期 | 模块如何被加载和管理 |
| 6 | 实现 `EventBus` | 发布/订阅 + 通配符匹配 | 模块间通信的基础设施 |
| 7 | 实现 `ConfigManager` | JSON 配置读写 + Schema 验证 | 统一配置管理 |
| 8 | 实现 `LogManager` | 双输出（文件 + 控制台）+ 日志轮转 | 统一日志管理 |
| 9 | 实现 `services/` 层 | GitHubService / PlaywrightService / FileSystemService / NetworkService | 外部依赖封装，模块不直接调 API |
| 10 | 实现 `utils/` 层 | shell / validators / formatters / constants | 无状态工具函数 |
| 11 | 实现 CLI 入口 | `yqa` 命令 + 子命令路由 | 统一的命令行入口 |
| 12 | 编写核心层单元测试 | `tests/core/` 全部覆盖 | 框架代码必须有测试 |
| 13 | 编写开发文档 | 架构设计 + 开发指南 + 模块模板 | 为自己和未来的你写文档 |

**M1.1 学习目标：**
- 理解抽象基类（ABC）和 `@abstractmethod` — 多态的基础
- 理解发布/订阅模式 — 解耦的核心手段
- 理解 Python 包结构和 `__init__.py` — 模块化基础
- 理解 `pyproject.toml` 和现代 Python 项目规范
- 理解 `dataclass` 和类型注解 — 数据建模
- 理解依赖注入 vs 直接导入 — 何时用哪种
- 理解 Service 层 vs Utils 层的区别 — 有状态 vs 无状态
- 理解日志系统的设计 — 级别、轮转、格式化
- 理解配置管理 — Schema 验证、默认值、环境覆盖

### M1.2: 核心模块 🎯

**预计时间**：2-3 周

**开发顺序（已确认）：Health → Backup → Update → New**

| # | 模块 | 核心功能 | 复用来源 | 为什么这个顺序 |
|---|------|----------|----------|----------------|
| 1 | **Health** | 10+ 项环境检查 + 健康评分 | `dev-check.sh` + `weekly-health.sh` | 最先做，因为它检查所有工具，后续模块依赖的工具都能被它验证 |
